# 開發路線圖

## MVP 範圍（先做這些）
- 首頁：本月當季精選＋分類入口＋搜尋
- 分類瀏覽＋搜尋
- 食材詳細頁（欄位見 DATA_SCHEMA.md），含品種切換（如適用）
- 依產季篩選「這個月可以買什麼」

## 先不做
- 帳號系統、我的最愛、社群、電商導購、食譜功能
- 拍照品質判斷（見下方「未來階段」）

## 開發階段
1. 資料內容：先完成 10–20 筆食材的完整資料，驗證欄位設計
2. App 專案：把已驗證的網頁雛形邏輯轉為 Flutter／React Native 專案
3. 內容擴充＋測試＋作品集呈現
4. （視情況）評估商業化方向

## 未來階段：拍照品質判斷
- 做法：不自建 CV 模型，改用具備視覺能力的 LLM（如 Claude API），照片＋該食材的挑選技巧文字一起送出，取得輔助判斷
- 限制：只有「用眼睛看得出來」的線索能判斷（顏色、光澤、外觀瑕疵），觸覺／嗅覺類重點（彈性、氣味）無法靠照片判斷，需在介面上誠實標示
- 安全注意：正式 App 需由後端代呼叫 API，不可把金鑰放在前端／App 內
- 現況：已完成可行性 demo（`prototypes/photo-quality-demo.jsx`），尚未決定實作時間點

## 費用量級參考（依 2026 年中公開牌價估算，僅供規劃參考）
- MVP 階段（自己與少量朋友測試）：後端／API 費用趨近 $0
- 拍照判斷功能（Claude API，以 Haiku 4.5 估算）：單次判斷約 US$0.003，主要成本隨使用量（使用者數 × 使用頻率）而非技術本身變動
- 上架費：Apple 開發者帳號約 US$99/年；Google Play 一次性 US$25
- 實際數字待有初步真實使用數據後再重新估算
